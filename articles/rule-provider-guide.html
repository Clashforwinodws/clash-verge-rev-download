📌 为什么需要 Rule Provider？

如果你曾经维护过 Clash 配置文件，一定对冗长的 rules 字段印象深刻。数百条 DOMAIN-SUFFIX、GEOIP 规则堆砌在主配置里，不仅让文件臃肿不堪，更新时更是噩梦：每次规则变动都需要手动编辑庞大的 YAML 文件，稍有不慎就会导致缩进错误，让整个代理系统瘫痪。

Rule Provider 的出现彻底改变了这一局面。它允许你将分流规则托管在远程 URL 或本地独立文件中，Clash Verge Rev 启动时自动拉取并合并这些规则集。这样一来，主配置文件只需要关注节点信息和策略组，规则部分变得极度精简。更棒的是，规则集可以独立更新，你甚至无需重启客户端就能获得最新的分流策略——这正是配置教程中强烈推荐的进阶用法。

💡 核心优势：主配置瘦身、规则集独立维护、社区规则一键引用、自动更新无感知。如果你还在手动复制粘贴规则，读完本文你会有全新的认识。

🧩 什么是 Rule Provider

Rule Provider 是 Mihomo 内核（Clash Meta）提供的一种规则加载机制。它本质上是一个指向外部规则文件的引用，支持两种来源：

  • HTTP(S) 远程加载：规则文件托管在 GitHub、CDN 或你的私人服务器上，Clash 定期检查更新。
  • 本地文件：直接读取本地磁盘上的 .yaml 或 .txt 规则文件。

一个典型的 Rule Provider 定义如下：

rule-providers: my_custom_rules: type: http url: "https://example.com/clash/rules.yaml" path: ./rules/custom.yaml interval: 86400 # 每天更新一次

定义好后，在规则列表中通过 RULE-SET 指令引用，就像调用一个函数一样。这极大提升了配置文件的可维护性，也与 Clash Verge Rev 的 Rule 模式 完美契合。

🛠️ 快速配置指南

步骤一：选择合适的规则源

社区有许多维护良好的规则集项目，例如 Loyalsoldier/clash-rules、ACL4SSR/ACL4SSR 等。这些项目将规则分类整理为 cn.yaml（国内直连）、global.yaml（全球代理）、ads.yaml（广告拦截）等多个文件，只需找到对应的 Raw 链接即可。当然，你也可以根据自己的需求定制规则文件。

步骤二：在配置文件中定义 Rule Provider

打开你的 Clash Verge Rev 配置文件（或直接在“配置”页面编辑），在顶层添加 rule-providers 字段：

rule-providers: cn_domains: type: http url: "https://cdn.jsdelivr.net/gh/Loyalsoldier/clash-rules@release/cn.yaml" path: ./rules/cn.yaml interval: 86400 global_domains: type: http url: "https://cdn.jsdelivr.net/gh/Loyalsoldier/clash-rules@release/proxy.yaml" path: ./rules/proxy.yaml interval: 86400
⚠️ 注意： 如果你是国内网络环境，GitHub 直连可能失败。建议使用 jsDelivr 等 CDN 加速链接，或者将规则文件托管在可访问的私有服务器上。

步骤三：在规则列表中引用

在 rules 字段中，用 RULE-SET 指令替代原来冗长的规则列表：

rules: - RULE-SET,cn_domains,DIRECT - RULE-SET,global_domains,🚀 代理 - GEOIP,,CN,DIRECT - MATCH,🚀 代理

保存配置后，Clash Verge Rev 会自动下载规则文件到本地 rules 目录，并根据你设置的间隔时间定期更新。从此，主配置文件从几千行锐减至几十行，清爽无比。

🌐 实战案例：国内外流量智能分流

下面是一个完整的 Rule Provider 实战配置，实现了国内网站直连、国外网站代理、广告域名屏蔽的功能。你可以直接复制到你的配置文件中使用。

# 1. 定义规则提供者 rule-providers: cn_domains: type: http url: "https://cdn.jsdelivr.net/gh/Loyalsoldier/clash-rules@release/cn.yaml" path: ./rules/cn.yaml interval: 86400 proxy_domains: type: http url: "https://cdn.jsdelivr.net/gh/Loyalsoldier/clash-rules@release/proxy.yaml" path: ./rules/proxy.yaml interval: 86400 ad_domains: type: http url: "https://cdn.jsdelivr.net/gh/privacy-protection-tools/anti-AD@master/anti-ad-clash.yaml" path: ./rules/ads.yaml interval: 86400 # 2. 规则匹配（从上到下优先） rules: - RULE-SET,ad_domains,REJECT - RULE-SET,cn_domains,DIRECT - RULE-SET,proxy_domains,🚀 自动选择 - GEOIP,CN,DIRECT - MATCH,🚀 自动选择

该配置会优先拦截广告域名，然后将国内域名直连，国外域名走代理。最后用 GEOIP 作为补充，兜底规则保证没有遗漏。配合 DNS 优化 和 Fake‑IP 模式，可以实现近乎完美的流量调度。

🚀 进阶技巧与最佳实践

合并多个规则集

你可以定义多个 Rule Provider，并在规则列表中按需组合。例如将广告规则、国内规则、流媒体规则分开维护，互不干扰。通过调整引用顺序，轻松控制匹配优先级。

使用自建规则集

如果你有特殊需求（例如内网域名、公司 VPN 路由），完全可以编写自己的 my-rules.yaml，放在 GitHub 私有仓库或本地服务器上。格式非常简单，每行一条规则：

# my-rules.yaml DOMAIN-SUFFIX,office.local,DIRECT IP-CIDR,10.0.0.0/8,DIRECT DOMAIN-KEYWORD,vpn,🚀 代理

然后在 rule-providers 中指向该文件即可。这比直接写在主配置中更加模块化，也便于团队共享。

监控规则更新状态

在 Clash Verge Rev 的日志页面，你可以看到每个 Rule Provider 的更新状态。如果出现“下载失败”，检查 URL 是否可访问，或者切换到备用 CDN。部分用户喜欢将规则文件缓存在本地，以避免网络波动影响代理启动——只需将 type 改为 file 并指定本地路径即可。

❓ 常见问题排查

Q: Rule Provider 下载失败怎么办？
检查网络连通性，尝试在浏览器中打开规则文件 URL。如果 URL 被墙，替换为可访问的 CDN 地址。也可以下载规则文件到本地，改用 type: file 加载。

Q: 修改规则后为什么没生效？
Rule Provider 会缓存规则文件到本地，默认每天更新一次。如需立即更新，可以删除缓存目录（通常为配置目录下的 rules 文件夹）并重启客户端，或者在图形界面中强制更新。

Q: 规则集和主配置规则冲突怎么办？
记住 Clash 的规则匹配是从上到下的。通常将 RULE-SET 放在前面，特殊单条规则紧随其后。如果你的自定义规则需要优先，就把它放在 RULE-SET 之上。更多细节可参考 Rule 模式详解。
